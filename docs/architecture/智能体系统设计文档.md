## 🎯 设计目标

基于 LangGraph 框架，构建一个**高效、创新、自适应**的多智能体协作系统，用于辅助学生学习高等数学。

### 核心设计原则

1. **神经符号双脑协同** - LLM（语义理解）+ SymPy（精确计算）
2. **流式实时反馈** - 降低首字节时间，提升用户体验
3. **情感感知教学** - 检测学生情绪，动态调整教学策略
4. **元认知反思** - 评估理解深度，防止机械记忆
5. **并行高效执行** - 多策略并行求解，竞速机制

---

## 🚀 核心创新点

### 1. 流式状态机架构（Streaming State Machine）

**问题**：传统的"请求-响应"模式延迟高，用户体验差。

**创新方案**：采用流式处理，实时推送智能体的思考过程。

```python
class StreamingState(TypedDict):
    """流式状态定义"""
    session_id: str
    student_id: str

    # 流式消息队列（支持实时推送）
    message_stream: Annotated[list, operator.add]

    # 智能体协作状态
    active_agents: set[str]  # 当前活跃的智能体
    agent_outputs: dict[str, Any]  # 各智能体的输出缓存

    # 学生认知状态（实时更新）
    cognitive_state: dict[str, float]  # {concept_id: mastery_prob}
    attention_span: float  # 注意力衰减系数（0-1）
    frustration_level: float  # 挫败感指数（0-1）
```

**优势**：
- ✅ 前端实时显示智能体思考过程（类似 ChatGPT 打字效果）
- ✅ 支持中断和恢复（学生可随时打断）
- ✅ 降低 TTFB（首字节时间），提升响应速度

---

### 2. 并行智能体编排（Parallel Agent Orchestration）

**问题**：现有设计是串行路由（Orchestrator → 单个 Agent），效率低。

**创新方案**：多个智能体并行工作，最优结果胜出。

```python
async def parallel_solve_node(state: StreamingState):
    """并行求解节点"""
    problem = state["current_problem"]

    # 同时启动多个求解策略
    tasks = [
        solver_sympy.solve(problem),      # SymPy 符号求解
        solver_numerical.solve(problem),  # 数值方法
        solver_llm.solve(problem),        # LLM 推理（备选）
    ]

    # 竞速：谁先返回正确结果谁胜出
    results = await asyncio.gather(*tasks, return_exceptions=True)

    # 验证和选择最优解
    best_result = select_best_solution(results)

    return best_result
```

**优势**：
- ✅ 提高求解成功率（多策略冗余）
- ✅ 降低延迟（竞速机制）
- ✅ 自动容错（某个智能体失败不影响整体）

---

### 3. 情感感知层（Affective Computing Layer）

**问题**：现有架构缺失情感检测，无法感知学生的挫败感和困惑。

**创新方案**：新增 **Emotion Detector** 智能体，实时分析学生情绪。

**检测信号**：
- 📝 **文本信号** - 情感词汇分析（"不懂"、"太难了"）
- ⏱️ **行为信号** - 响应时间、连续错误次数、求助频率
- 📊 **学习模式** - 任务停留时间、尝试次数

**干预策略**：
- 😤 **挫败（Frustrated）** → 降低难度，提供鼓励
- 😕 **困惑（Confused）** → 换个角度解释，提供示例
- 😎 **自信（Confident）** → 增加挑战，推送进阶题目
- 😴 **无聊（Bored）** → 引入趣味性内容，改变教学方式

---

### 4. 元认知反思层（Metacognitive Reflection）

**问题**：学生可能只是机械记忆，而非真正理解。

**创新方案**：新增 **Reflection Agent**，评估理解深度。

**检测方法**：
1. **变式问题测试** - 改变问题条件，观察学生是否能迁移
2. **解释性提问** - "为什么这一步这样做？"
3. **反例测试** - 提供错误示例，看学生能否识别

**示例**：
```python
# 学生刚做对一道导数题
reflection_agent.ask_deep_question(
    "如果函数不可导，这个方法还适用吗？为什么？"
)

# 如果学生回答不出，说明只是记住了步骤，未真正理解
# → 触发概念回顾节点
```

---

### 5. 自适应提示工程（Adaptive Prompting）

**问题**：不同学生需要不同的教学风格，固定 Prompt 无法满足。

**创新方案**：根据学生画像动态生成 Prompt。

**适配维度**：
- 🎨 **学习风格** - 视觉型（多用图表）vs 语言型（详细描述）
- 📚 **知识水平** - 初学者（简单语言）vs 进阶者（严谨术语）
- 😊 **情绪状态** - 挫败时多鼓励，自信时增加挑战

**示例**：
```python
# 视觉型学习者 + 初学者 + 有挫败感
prompt = """
你是一位耐心的数学导师。
- 优先使用图形和几何直觉来解释概念
- 使用简单的语言，避免专业术语
- 学生当前有些挫败，请多鼓励，降低难度
"""

# 语言型学习者 + 进阶者 + 自信
prompt = """
你是一位严谨的数学教授。
- 使用详细的文字描述和逻辑推导
- 可以使用更严谨的数学语言和符号
- 学生基础扎实，可以增加挑战性
"""
```

### 6. 验证器机制（Verifier）⭐ v1.1 新增

**问题**：并行求解的核心难点不是"并行"，而是"如何判定谁是正确的/更好的"。

**创新方案**：引入显式的 **Verifier（验证/裁判）** 能力，对候选解做可解释、可复现的判定。

**Verifier 职责**：
- ✅ **等价性验证** - SymPy `simplify`/`equals` + 约束/假设（定义域、参数条件）
- ✅ **数值回代** - 在合法域内抽样验证（避免符号等价失败或过慢）
- ✅ **约束校验** - 初值/边界条件、解的分支选择、是否遗漏常数项
- ✅ **步骤合法性检查** - 每步变形是否可由可枚举规则解释

**输出格式**：
```python
class VerifierOutput(TypedDict):
    is_valid: bool              # 是否有效
    confidence: float           # 置信度 0-1
    failure_reason: str | None  # domain_error | missing_constant | non_equivalent | ambiguity | timeout
    normalized_answer: str      # 统一格式的答案表示（便于缓存与去重）
```

---

### 7. 状态与事件分层（State/Event Separation）⭐ v1.1 新增

**问题**：当前 `StreamingState` 同时承担业务状态与展示事件，存在状态膨胀、难以回放/审计的问题。

**创新方案**：明确两层分离设计。

| 层级 | 内容 | 持久化策略 |
|------|------|-----------|
| **Durable State（持久层）** | 认知状态、路径位置、关键诊断结论、上下文摘要 | 强制持久化 |
| **Event Stream（事件流）** | thinking/tool_call/hint/deep_question/error | 可选落盘 |

**收益**：
- ✅ 更容易做"重放（replay）"与质量复盘
- ✅ 更容易做性能优化（限长、采样、按事件类型落库）
- ✅ 更容易做合规（敏感事件可脱敏/不落库）

---

### 8. 策略控制器（Policy Controller）⭐ v1.1 新增

**问题**：当前编排逻辑主要依赖意图分类 + 固定路由，缺少成本、延迟、可靠性的考量。

**创新方案**：Orchestrator 升级为"策略控制器"，引入预算概念。

**策略参数**：
```python
class PolicyConfig(TypedDict):
    risk_level: Literal["low", "medium", "high"]  # 风险等级
    latency_budget_ms: int      # 本轮最大可用时延
    token_budget: int           # 本轮最大 token
    tool_budget: int            # 本轮最多工具调用次数
    parallelism: int            # 本轮允许的并行度
```

**Fast-Path / Slow-Path 路由**：
- 🚀 **Fast-Path**（高置信、低风险）→ 缓存/规则/小模型/单 Agent
- 🐢 **Slow-Path**（低置信、高风险）→ 多 Agent 并行 + Verifier + 更强模型

## 🧩 智能体角色定义

### 现有智能体（优化）

| 智能体 | 职责 | 创新点 |
|--------|------|--------|
| **Orchestrator** | 意图识别、任务路由 | 支持流式状态管理 |
| **Solver** | 数学问题求解 | 并行多策略求解 |
| **Diagnostician** | 错误诊断 | OCR + 步骤比对 |
| **Tutor** | 教学交互 | 自适应 Prompt |
| **Planner** | 学习路径规划 | DKT + 知识图谱 |

### 新增智能体（创新）

| 智能体 | 职责 | 价值 |
|--------|------|------|
| **Emotion Detector** | 情感检测与干预 | 提升学习体验，降低挫败感 |
| **Reflection Agent** | 元认知评估 | 确保真正理解，防止机械记忆 |
| **Verifier** ⭐ | 答案验证与裁判 | 确保并行求解的正确性判定 |
| **Safety Module** ⭐ | 安全过滤与合规 | 代码执行风险、敏感信息过滤 |

## 📊 评测体系矩阵 ⭐ v1.1 新增

### 质量评测维度

| 智能体 | 评测维度 | 关键指标 |
|--------|---------|---------|
| **Solver** | 题型覆盖、等价性验证通过率、步骤质量 | 求解成功率 > 95%，失败原因分布 |
| **Diagnostician** | 错误定位偏差、错误类型混淆矩阵 | 定位准确率 > 85%，提示后纠错率 |
| **Tutor** | 学习增益（前测/后测）、提示依赖度 | 迁移题提升率、独立完成率 |
| **Emotion** | 误伤率（不该干预却干预） | 干预有效率 > 70% |
| **Reflection** | 校准改善（自信与表现差距缩小） | 机械记忆检出率 |
| **Verifier** | 验证准确率、误判率 | 等价性判定准确率 > 98% |

### 评测资产建议

- 📝 **种子题集**：每类 20-50 题 + 标准解（可验证）
- ❌ **常见错误样例库**：按错误类型分类
- 📊 **学习增益基准**：前测/后测对照数据

## 📊 技术选型

### 核心框架
- **LangGraph** `>=0.2.0` - 多智能体编排
- **LangChain** `>=0.3.0` - LLM 抽象层
- **FastAPI** `>=0.115.0` - 异步 Web 框架

### AI 模型
- **LLM**: DeepSeek-V3 / Qwen-2.5-Math（推理）
- **Embedding**: BGE-M3（数学语料微调）
- **OCR**: Texify（数学公式识别）

### 计算引擎
- **SymPy** `>=1.13.0` - 符号计算
- **NumPy** `>=1.26.0` - 数值计算

### 数据存储
- **PostgreSQL** - 业务数据（学生、练习、会话）
- **Redis** - 缓存（求解结果、学生画像）
- **Neo4j** - 知识图谱
- **Qdrant** - 向量检索（HybridRAG）

### 代码执行
- **E2B Code Interpreter** - 云端代码沙箱（推荐）
- **Docker** - 本地沙箱（备选）

## 📈 性能指标

### 目标性能

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **首字节时间 (TTFB)** | < 500ms | 流式响应开始时间 |
| **求解延迟** | < 2s | 数学问题求解时间 |
| **并发支持** | 1000+ | 同时在线学生数 |
| **缓存命中率** | > 80% | 常见问题缓存命中 |
| **智能体切换延迟** | < 100ms | LangGraph 节点切换 |

### 优化策略
- ✅ Redis 缓存（求解结果、学生画像）
- ✅ 批量推理（vLLM 批量处理）
- ✅ 预测性预加载（提前加载下一个知识点）
- ✅ 并行执行（多策略竞速）
