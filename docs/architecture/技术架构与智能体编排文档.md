## 三、智能体系统架构

### 3.1 核心设计理念

#### 神经符号双脑协同 (Neuro-Symbolic Synergy)

```
                    ┌──────────────────┐
                    │    用户输入       │
                    └────────┬─────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
    ┌─────────▼──────┐       │    ┌─────────▼──────┐
    │   右脑 (LLM)    │       │    │  左脑 (SymPy)   │
    │                │       │    │                │
    │ · 语义理解      │       │    │ · 符号计算       │
    │ · 教学话术      │       │    │ · 真值验证       │
    │ · 情感支持      │       │    │ · 等价性检查     │
    │ · 苏格拉底引导   │       │    │ · 数值采样       │
    └────────────────┘       │    └────────────────┘
              │              │              │
              └──────────────┼──────────────┘
                             │
                    ┌────────▼─────────┐
                    │  消除计算"幻觉"    │
                    │  精确+自然的教学   │
                    └──────────────────┘
```

### 3.2 智能体列表

经过重构，从 9 个智能体精简为 **4 个核心智能体 + 1 个纯函数路由**：

| 智能体 | 类型 | LLM 调用 | 职责 |
|--------|------|----------|------|
| **Router** | 纯函数 | 0 次 | 关键词意图识别 → 任务分发 |
| **MathSolver** | 内容智能体 | 2 次 | SymPy 代码生成 + 解题步骤生成 |
| **Tutor** | 内容智能体 | 1 次 | 概念讲解 / 苏格拉底式教学 / 学习规划 |
| **Diagnostician** | 内容智能体 | 1 次 | 答案诊断 / 步骤比对 / 错误分类 |
| **Tracker** | 追踪智能体 | 0 次 | 学习数据追踪，更新掌握度向量 |

### 3.3 智能体工作流 (LangGraph)

#### 3.3.1 工作流拓扑

```
START
  │
  ▼
entry (初始化 + 纯函数路由)
  │
  ├── intent=SOLVE ──────────► math_solver ──┐
  │                                          │
  ├── intent=TEACH/PLAN ─────► tutor ────────┤
  │                                          │
  └── intent=DIAGNOSE ───────► diagnostician ─┤
                                             │
                                     ┌───────┘
                                     ▼
                                 tracker (零 LLM，纯计算)
                                     │
                                     ▼
                                    END
```

#### 3.3.2 工作流状态 (StreamingState)

工作流使用 `TypedDict` 定义全局状态，通过 `Annotated` 类型支持增量更新：

```python
class StreamingState(TypedDict, total=False):
    # 会话标识
    session_id: str
    student_id: str

    # 消息流 (operator.add 自动追加)
    message_stream: Annotated[list[dict], operator.add]

    # 路由控制
    intent: str | None          # solve / teach / diagnose / plan
    tutor_mode: str | None      # teach / plan

    # 教学上下文
    last_message: str
    current_concept: str | None
    current_problem: str | None
    interaction_history: list[dict]
    attachments: list[str]

    # 学生上下文 (entry 时从 DB 加载)
    student_context: dict       # mastery_vector / error_tendency 等
    consecutive_errors: int

    # 智能体输出 (merge_dicts 合并)
    agent_outputs: Annotated[dict, merge_dicts]

    # 追踪数据
    tracking_data: dict         # 本次交互的追踪数据
    profile_updates: dict       # Tracker 生成的画像更新指令

    # 元数据
    session_start_time: float
    total_interactions: int
```

#### 3.3.3 检查点持久化

- **生产环境**: Redis 检查点 (`RedisCheckpointer`, TTL=3600s)
- **开发环境**: 内存检查点 (`MemorySaver`)
- **降级策略**: Redis 初始化失败时自动回退到内存检查点

### 3.4 智能体编排流程详解

#### 3.4.1 Entry 节点（入口）

```
用户消息 + 附件
       │
       ▼
  entry_node()
       │
       ├── 1. 初始化会话时间
       ├── 2. 提取消息和附件
       └── 3. 纯函数意图分类 classify_intent()
              │
              ├── 有附件 → DIAGNOSE
              ├── 关键词匹配 → SOLVE / TEACH / DIAGNOSE / PLAN
              ├── 数学表达式检测 → SOLVE
              └── 默认 → TEACH
```

**意图路由映射表**:

| 意图 | 关键词示例 | 目标节点 |
|------|-----------|---------|
| SOLVE | 计算/求解/求导/积分/解方程 | math_solver |
| TEACH | 什么是/定义/解释/为什么/你好 | tutor |
| DIAGNOSE | 我的答案/对不对/批改/检查 | diagnostician |
| PLAN | 学习路径/怎么学/练习/出一道题 | tutor (PLAN 模式) |

#### 3.4.2 MathSolver 节点（数学求解器）

```
用户问题
   │
   ▼
┌──────────────────────────────────────────────┐
│ MathSolverAgent                               │
│                                               │
│  Step 1: 问题缓存查找 (hash_problem)           │
│     ├── 命中缓存 → 直接返回                     │
│     └── 未命中 → 继续                           │
│                                               │
│  Step 2: LLM 调用 #1 — SymPy 代码生成          │
│     · System Prompt: 数学→Python/SymPy 转换     │
│     · 输出: 可执行的 SymPy 代码                  │
│                                               │
│  Step 3: 代码安全检查 (纯正则，零 LLM)           │
│     · 检查危险模块导入                           │
│     · 检查文件系统操作                           │
│     · 检查网络操作                              │
│                                               │
│  Step 4: SymPy 代码执行                         │
│     · 在沙箱环境中执行                           │
│     · 获取计算结果 result                        │
│                                               │
│  Step 5: 结果验证 (SymPy 等价性检查，零 LLM)     │
│     · Layer 1: 标准化字符串比较                   │
│     · Layer 2: SymPy 符号等价                    │
│     · Layer 3: 数值采样验证                      │
│     · Layer 4: LLM 辅助判断 (仅必要时)            │
│                                               │
│  Step 6: LLM 调用 #2 — 解题步骤生成              │
│     · 生成人类可读的详细解题步骤                   │
│     · 使用 KaTeX 兼容的数学公式                   │
│                                               │
│  Step 7: 流式输出 (StreamWriter)                │
│     · 逐 token 推送到前端                        │
└──────────────────────────────────────────────┘
```

#### 3.4.5 Tracker 节点（学习追踪器）

```
tracking_data (从前序智能体输出)
   │
   ▼
┌──────────────────────────────────────────────┐
│ TrackerAgent (零 LLM 调用)                     │
│                                               │
│  Step 1: 更新 mastery_vector (掌握度向量)       │
│     · 简化贝叶斯更新公式:                        │
│       new = old + α × (outcome - old)          │
│       α = BASE_ALPHA × (1 + difficulty)        │
│     · outcome = 1.0(正确) / 0.0(错误)           │
│                                               │
│  Step 2: 更新 error_tendency (错误倾向)         │
│     · 按错误类型累加计数                          │
│                                               │
│  Step 3: 更新 recent_concepts                  │
│     · 维护最近学习的 20 个知识点                   │
│     · 最近使用的排在前面                          │
│                                               │
│  Step 4: 更新练习计数                            │
│     · total_exercises_delta                    │
│     · correct_count_delta                      │
│                                               │
│  Step 5: 更新 consecutive_errors               │
│     · 正确 → 归零                               │
│     · 错误 → +1                                │
│                                               │
│  输出: profile_updates → session_service 写回 DB │
└──────────────────────────────────────────────┘
```

### 3.5 数学等价性验证引擎

位于 `app/agents/core/math_equivalence.py`，采用多层级验证策略：

```
学生答案 vs 标准答案
         │
         ▼
Layer 1: 标准化字符串比较 ──→ 匹配 → ✅ 等价
         │ 不匹配
         ▼
Layer 2: SymPy 符号等价 ────→ 匹配 → ✅ 等价
         │ (simplify/trigsimp/expand/factor)
         │ 不匹配
         ▼
Layer 3: 数值采样验证 ──────→ 匹配 → ✅ 等价
         │ (多点随机采样比较)
         │ 不匹配
         ▼
Layer 4: LLM 辅助判断 ──────→ 结果
         (仅前三层无法确定时，1次 LLM)
```

- Layer 1-3 均为零 LLM 调用，纯计算
- SymPy 计算使用独立进程池 (`ProcessPoolExecutor`, max_workers=2)，避免 GIL 阻塞

## 四、贝叶斯知识追踪 (BKT)

### 4.1 BKT 模型

BKT 用于实时预测学生对每个知识点的掌握度：

```
参数：
  P(L₀)  — 初始掌握概率
  P(T)   — 学习率（每次交互后学会的概率）
  P(G)   — 猜测率（不会但答对的概率）
  P(S)   — 失误率（会但答错的概率）

更新公式 (Tracker 简化版):
  new_mastery = old_mastery + α × (outcome - old_mastery)
  α = 0.1 × (1 + difficulty)
```

### 4.2 BKT 服务 (`BKTService`)

- **缓存策略**: 内存缓存 (L1) → 数据库 (L2)
- **Prometheus 指标**: 记录每次 BKT 更新耗时
- **遗忘模型**: `apply_forgetting()` 处理长时间未学习的掌握度衰减
